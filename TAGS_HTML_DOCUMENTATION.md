# SMX HTML Tag Documentation

## Purpose

This document explains how HTML-based ad tags are generated in SMX today, what kind of tags the platform emits, how DSP macros are applied, and which parts are standard-market patterns versus formal IAB standards.

It is written for developers and operators who need to understand:

- how display tags are formed
- how the backend serves HTML creatives
- what role DSP selection plays
- which macros are supported
- what example outputs should look like

---

## Executive Summary

The current implementation separates two independent concerns:

1. `Tag shape`
   - The structural form of the snippet delivered to the publisher.
   - Examples:
     - `display-js`
     - `display-iframe`
     - `display-ins`
     - `native-js`
     - `tracker-click`
     - `tracker-impression`

2. `DSP macros`
   - Basis, Illumin, TTD, DV360, Xandr.
   - DSP selection only injects trafficking/tracking macros where appropriate.
   - DSP selection does **not** change the HTML tag shape.

This separation is intentional and enforced by code.

Correct rule:

- `tag type` = product choice
- `DSP` = macro layer only

## High-Level Flow

### 1. Tag Builder chooses a snippet variant

The frontend Tag Builder lets the user choose a snippet variant such as:

- `display-js`
- `display-iframe`
- `display-ins`
- `native-js`
- `tracker-click`
- `tracker-impression`

For display creatives, the default snippet is:

- `display-js`

This selection happens in:

- [apps/web/src/tags/TagSnippetPanel.tsx](/Users/enzocienfuegos/Documents/MandaRion/apps/web/src/tags/TagSnippetPanel.tsx)

### 2. The snippet is generated by a pure function

The frontend calls:

```js
buildTagSnippet(tag, variant, servingBaseUrl, campaignDsp, diagnostics)
```

That function lives in:

- [packages/contracts/src/tag-snippets.mjs](/Users/enzocienfuegos/Documents/MandaRion/packages/contracts/src/tag-snippets.mjs)

This function:

- builds serving URLs
- builds tracker URLs
- applies DSP macros where appropriate
- returns the final snippet string

### 3. The publisher places the snippet

Depending on the selected tag type, the publisher page will load:

- a JS loader
- a direct iframe
- an INS placeholder + bootstrap script
- a tracking URL only

### 4. The API serves the creative wrapper

The serving endpoints are implemented in:

- [apps/api/src/modules/adserver/display/routes.mjs](/Users/enzocienfuegos/Documents/MandaRion/apps/api/src/modules/adserver/display/routes.mjs)

Key public routes:

- `GET /v1/tags/display/:id.html`
- `GET /v1/tags/display/:id.js`
- `GET /v1/tags/native/:id.js`

The API:

- resolves the active creative bound to the tag
- returns an HTML wrapper or JS loader
- loads the creative from its published `public_url`
- fires impression tracking
- bridges click tracking via `postMessage`

## Important Rule: DSP Does Not Change HTML Tag Shape

This is the most important behavioral rule.

### Correct behavior

DSP selection:

- `basis`
- `illumin`
- `ttd`
- `dv360`
- `xandr`

only affects:

- tracker URLs
- trafficking/VAST URLs where applicable

DSP selection does **not** affect:

- whether the tag is `display-js`
- whether the tag is `display-iframe`
- whether the tag is `display-ins`

### Why this rule exists

Because DSP selection is a trafficking concern, not a rendering concern.

The platform must not do this:

- `DSP = Basis` => emit a special runtime blob
- `DSP = Illumin` => change the display tag structure

That behavior was considered incorrect and has been removed.

### Current enforced behavior

For HTML/display tags:

- `display-js` always returns the standard JS tag
- `display-iframe` always returns the standard iframe tag
- `display-ins` always returns the standard INS tag

For tracker variants:

- `tracker-click` always returns a click URL
- `tracker-impression` always returns an impression URL

DSP only modifies macro parameters on those URLs.

## Supported HTML Tag Types

## 1. `display-js`

### Purpose

Standard JavaScript display tag for publisher pages.

### How it works

The tag loads:

- `/v1/tags/display/:id.js`

That JS loader:

- finds `document.currentScript`
- creates an iframe
- points the iframe to `/v1/tags/display/:id.html`
- replaces the script element in the DOM

### Generated by

- `buildDisplayJsSnippet(...)`
in
- [packages/contracts/src/tag-snippets.mjs](/Users/enzocienfuegos/Documents/MandaRion/packages/contracts/src/tag-snippets.mjs)

### Example

```html
<script src="https://api-staging.duskplatform.co/v1/tags/display/521e46b5-c702-4165-89c3-5bec5ff1358d.js" async></script>
<noscript>
  <iframe
    src="https://api-staging.duskplatform.co/v1/tags/display/521e46b5-c702-4165-89c3-5bec5ff1358d.html"
    width="300"
    height="250"
    scrolling="no"
    frameborder="0"
    style="border:0;overflow:hidden;"
  ></iframe>
</noscript>
```

### Notes

- The serving URL stays clean.
- No DSP macros belong on the `.js` or `.html` serving URL.

## 3. `display-ins`

### Purpose

Slot placeholder pattern using an `<ins>` container and inline bootstrap script.

### How it works

The platform emits:

- an `<ins>` placeholder
- a small script that mounts an iframe into that slot

It includes a `MutationObserver` fallback in case the slot appears later in the DOM lifecycle.

### Generated by

- `buildDisplayInsSnippet(...)`
in
- [packages/contracts/src/tag-snippets.mjs](/Users/enzocienfuegos/Documents/MandaRion/packages/contracts/src/tag-snippets.mjs)

### Example

```html
<ins id="smx-ad-slot-521e46b5-c702-4165-89c3-5bec5ff1358d" style="display:inline-block;width:300px;height:250px;"></ins>
<script>
(function(){
  var SLOT_ID = "smx-ad-slot-521e46b5-c702-4165-89c3-5bec5ff1358d";
  var SRC     = "https://api-staging.duskplatform.co/v1/tags/display/521e46b5-c702-4165-89c3-5bec5ff1358d.html";
  var W = '300'; var H = '250';

  function mount(slot) {
    if (!slot || slot.dataset.smxMounted) return;
    slot.dataset.smxMounted = '1';
    var iframe = document.createElement('iframe');
    iframe.src = SRC;
    iframe.width = W;
    iframe.height = H;
    iframe.scrolling = 'no';
    iframe.frameBorder = '0';
    iframe.style.cssText = 'border:0;overflow:hidden;display:block;';
    iframe.setAttribute('sandbox','allow-scripts allow-same-origin allow-popups allow-forms allow-top-navigation-by-user-activation');
    slot.replaceWith(iframe);
    if (observer) observer.disconnect();
  }

  var slot = document.getElementById(SLOT_ID);
  if (slot) { mount(slot); return; }

  var observer = typeof MutationObserver === 'function'
    ? new MutationObserver(function() { mount(document.getElementById(SLOT_ID)); })
    : null;
  if (observer) observer.observe(document.body || document.documentElement, { childList: true, subtree: true });
})();
</script>
```

### Notes

- Useful for placements expecting a placeholder node.
- Still ultimately renders through an iframe to the SMX wrapper.

## How HTML Creatives Are Actually Served

The display serving backend is implemented in:

- [apps/api/src/modules/adserver/display/routes.mjs](/Users/enzocienfuegos/Documents/MandaRion/apps/api/src/modules/adserver/display/routes.mjs)

## Public serving routes

- `GET /v1/tags/display/:id.html`
- `GET /v1/tags/display/:id.js`
- `GET /v1/tags/native/:id.js`

## Route behavior

### `/v1/tags/display/:id.html`

This route:

1. validates `tagId`
2. loads the active tag and active creative binding
3. resolves the published `creative_versions.public_url`
4. builds an HTML wrapper
5. embeds the creative in an iframe
6. optionally fires impression tracking
7. tracks clicks using a `postMessage` bridge

### `/v1/tags/display/:id.js`

This route returns a JavaScript loader that:

- creates an iframe
- points it to `/v1/tags/display/:id.html`
- mounts it where the script tag was placed

### `/v1/tags/native/:id.js`

Currently behaves as an alias-style native loader entry point.

---

## Wrapper HTML Behavior

The HTML wrapper produced by `buildDisplayHtml(...)` does the following:

- emits a fixed-size page
- loads the creative via `<iframe src="{creative public_url}">`
- fires impression via `impression.gif`
- listens for:

```js
{ type: 'smx:exit', url: 'https://destination.example.com' }
```

- then:
  - fires click tracking
  - redirects top window or opens a new tab

### Important note

The wrapper is an SMX serving shell around a published creative bundle.

That means:

- the tag does not inline the creative HTML itself
- it loads a wrapper
- and the wrapper loads the published creative asset

## Are These Tags IAB Standard?

## Short answer

Partially.

## More precise answer

### Yes, in the sense of common market patterns

These are normal market-compatible patterns:

- JS ad tag that mounts an iframe
- iframe embed tag
- slot placeholder + bootstrap tag
- tracking pixel URL
- click tracker URL

### Yes, formally, for video-related standards

For video and measurement:

- VAST is an IAB standard
- OMID is part of IAB measurement standards

### No, not as a single formal “IAB HTML display tag spec”

There is no single universal “IAB HTML display tag” format in the same way VAST works for video.

So the correct documentation wording is:

- `display-js`, `display-iframe`, and `display-ins` are market-standard delivery patterns
- `VAST` and `OMID` align with IAB standards

Avoid documenting the HTML display tags as if they were a formal IAB spec artifact.

DSP macros are defined in:

- [packages/contracts/src/dsp-macros.mjs](/Users/enzocienfuegos/Documents/MandaRion/packages/contracts/src/dsp-macros.mjs)

Supported DSPs:

- Basis
- Illumin
- TTD
- DV360
- Xandr

## Important rule

Macros belong on:

- click trackers
- impression trackers
- engagement trackers
- trafficking/VAST URLs where applicable

Macros do **not** belong on:

- `/v1/tags/display/:id.js`
- `/v1/tags/display/:id.html`
- `/v1/tags/native/:id.js`

These serving URLs must remain clean for:

- correctness
- cacheability
- simplicity
- standards compatibility

## Non-Negotiable Rules for Future Development

1. Never let DSP selection change the HTML tag structure.
2. Keep serving URLs macro-free.
3. Only put DSP macros on tracking/trafficking URLs.
4. Keep HTML display delivery simple and iframe-based.
5. Treat VAST/OMID as standards-driven layers.
6. Treat HTML display tags as market-standard patterns, not as a formal single IAB display spec.

## Summary

For HTML creatives, the system now works like this:

- the product decides the tag shape
- the API serves a wrapper around a published creative
- the wrapper loads the creative in an iframe
- trackers handle impression/click/engagement
- DSP selection only injects macros

That means:

- `HTML tag shape` is stable
- `DSP` is only metadata/macro enrichment
- `serving` and `tracking` remain cleanly separated
